Proceso de Submit de Objetos Toba

Durante el transcurso de una transacción a nivel de operación el control se intercambia entre el cliente y el servidor. El momento en que se pasa del lado cliente al servidor es decidido por el proceso de submit desde el javascript de la página.

Formas de ir al servidor

La transición desde el cliente hacia el servidor se puede hacer a través de dos métodos, llamados GET y POST, el primero codifica la información en la misma URL mientras que el segundo envía la información en el cuerpo del pedido. La ventaja visible del método GET es que es sencillo hacer un bookmark del pedido (por ejemplo el pedido de un reporte) o enviar por correo la dirección y garantizar que van a observar exactamente la misma información. También tiene sus problemas que lo hacen poco viable para el tipo de problemas que se quieren resolver con Toba, por una lado posee un límite bastante pequeño para la cantidad de información que se puede enviar y por otro lado el medio por el cual se dispara el método GET es simplemente un link HTML con lo cual es imposible 'entrometerse' en este proceso.

El método POST posibilita que los distintos objeto-toba puedan formar parte de la desición de ir o no al servidor e incluir comportamientos una vez tomada esta decisión y es el método utilizado durante el transcurso de una operación.

Inicio del proceso

El procesa se inicia cuando un objeto-toba decide que para satisfacer un pedido del usuario necesita ir al servidor, generalmente con atrapando un evento onclick de algún tag HTML. Como todo objeto-toba tiene una clase javascript con la que se entiende, el pedido de cambio se le notifica a la instancia de esta clase.

¿Se puede hacer submit?

Salir del cliente no puede ser una decisión unilateral de un objeto, todos deben participar. Un objeto puede decir "yo no puedo" por ejemplo en caso de que sus datos no sean válidos, o que pidiendo una confirmación al cliente, este decida cancelar la operación. El responsable de controlar este proceso es el javascript objeto_ci de mayor nivel en la operación, una vez obtenida su afirmación busca la de todos sus objetos contenidos, y así sucesivamente. En caso de que cualquiera niegue la posibilidad de submit se retrocede el proceso y se continúa la ejecución en el cliente.

En caso de que todos los objetos contenidos en la pantalla respondan afirmativamente se les notifica que preparen sus campos de comunicación con el servidor (típicamente campos del formulario enviados en el POST).

Fin del proceso

El proceso termina cuando el CI de mayor nivel ejecuta el método submit del form en el que se encuentra casi toda la pantalla.
