Temas charlados

Elementos toba (API del sistema)

  • Distribuir responsabilidades de una forma adecuada las clases que interactuan para representar el elemento.
  • Una clase deberia englobar la creacion de un objeto por partes en forma dinamica (Esta idea tiene una relacion con "Enmascarar db_tablas con clases de dominio", en este caso la clase de dominio seria el elemento_toba comprometido).
  • El elemento_* es el responsable de devolver referencias de:
    • el db_tablas del elemento
    • el objeto runtime generado por una instanciacion del mismo
  • El objeto runtime no sabe nada de sus parametros (se pasan en el contructor)
  • Deberia existir un constuctor centralizado que despache preguntas a un elemento especifico

Comportamientos de pantalla recurrentes

  • Hay que proveer un esquema que resuelva estos comportamientos recurrentes del CI
    • dbr => form
    • dbr => form_ml
    • dbr => form_ml posicion
    • dbr => form + cuadro + $seleccion
    • dbr => form_ml + (ei)* + $seleccion
    • dbr => ci
    • array => form + cuadro + $seleccion
  • Este trabajo tiene que derivar en un catalogo de comportamientos

Personalizaciones

  • Se usarian ARCHIVOS PLANOS que representen metadatos del mayor nivel de atomicidad posible (se prefirio SQL sobre XML para la representacion) y SVN como medio de versionado distribucion.
  • Personalizacion en toba como ejemplo
    • hacer reportes
    • personalizarlos para algun proyecto
    • integrar las personalizaciones a un esquema que suponga al svn
    • iniciar una instancia con un set de personalizaciones
