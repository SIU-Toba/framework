Toba 0.9.0

  • Fecha: 21 de Febrero de 2006
  • Fuente: source:tags/0.9.0
  • Revisiones involucradas: [976] a [1269]
  • Plan de la versión
  • Instructivo de migración.

A continuación se muestran los cambios más significativos introducidos en esta versión.

Persistencia

  • Manejo de relaciones N-N (utilizando subselects para la carga y guardando la relación entre filas para la sincro)
  • Manejo de un cursor a nivel de tabla, afecta a todas las consultas publicas y a la inserción de nuevas filas.
  • Borrado en cascada en memoria (aún no parametrizable)
  • El orden de la sincronización y carga se detecta automáticamente, aunque se puede especificar uno particular ordenando las dependencias .
  • Modo debug tanto en definición como en runtime. Utiliza el nuevo ei_esquema
  • Se elimina el identificador (nombre) para las asociaciones entre tablas
  • Mejora estetica a las consultas SQL
  • El retraso de constraints es configurable por definición
  • La excepcion de "estructura incorrecta" cuando sobra algún campo se reemplaza por un warning en el logger.

Comandos administrativos

  • Administración multi-instancia.
  • Exportación granular de componentes.
  • Migración a PHP. Se modificó el layout de las carpetas. (Esta en php/consola y los comandos están en una subcarpeta que se llama comandos).
  • Compilación de metadatos de los proyectos.
  • Cambio del mecanismo de actualización de secuencias. Cada grupo de desarrollo de un mismo proyecto debería definir un número distinto en la entrada id_grupo_desarrollo del archivo instalación/instalacion.ini. Esto define un 'namespace' de un millón de números para cada grupo. Al regenerar la instancia o el proyecto las secuencias de cada grupo se actualizan en base a su 'namespace' particular, evitando que se choquen al juntar los metadatos. Si la constante no se define se comporta como antes, por lo que no tiene sentido que un grupo lo defina y otro no (siempre en un mismo proyecto). Ver Grupos de desarrollo.

Modificaciones en la arquitectura interna

  • Puertas de acceso al núcleo
  • Unificación del flujo de metadatos. Se abstrajo el mecanismo de carga y construcción de componentes. Pueden consumirse definiciones compiladas o provenientes de la DB en forma indistinta.
  • Los ítems pasan utilizar por defecto un nuevo tipo de solicitud: la web (solicitud_web), viene en reemplazo de la vieja solicitud_browser. Esta solicitud conoce la existencia de CI's y CN's, conoce a la zona y dos nuevos conceptos que giran en un pedido de página:
    • Los eventos que se atienden de la página anteriores (sería el $_POST)
    • Los servicios que se brindan a la nueva página (salida que produce) La nueva solicitud procesa todos los pedidos en este orden permitiendo tener procesos similares pero que brindan distintos servicios (el html usual, AJAX, cascadas, PDF, reportes, etc) pero que siguen teniendo acceso a la transacción en curso.
  • La nueva solicitud web no conoce los conceptos de: acción, buffer o patrón, estos quedan resumidos a la solicitud_browser.

Objetos Varios

  • Nuevo tipo de tooltip que permite links y html, a diferencia del anterior no se mueve con el mouse y permite que se pueda seguir un link contenido.
  • Las descripciones de los tooltips pueden contener tags del tipo [wiki:Id Descripción] o [api:Id Descripción] apuntando al wiki y al api offline.
  • Los objetos que están colapsados desde el server van en ese estado al cliente (ya no se hace por javascript una vez cargada la pagina)
  • En CI: Inclusión del método $this->dependencia($id) que reemplaza al acceso directo a $this->dependencias[$id] cargando bajo demanda a un objeto
  • Cambios en la api del solo_lectura:
    • Los metodos pasan a llamarse set_solo_lectura(true/false) tanto para el cliente como para el servidor
    • Se soporta solo_lectura en cliente para ef_multi_seleccion_check
    • Los editables con mascara respetan el solo_lectura
    • El ef_popup y el ef_editable_fecha ocultan su vinculo
    • Los editables en general usan readOnly en lugar de disabled (se comporta mejor para textarea y además se envía por POST)
    • Modificación del comportamiento del ef_upload, en modo solo_lectura solo muestra la etiqueta. En caso de ser modificable muestra el componente si no se especifica estado y muestra componente + elección de modificación en caso contrario.
  • El estado de un ef_upload en el carga puede ser nulo o el nombre del archivo (antes esperaba un arreglo para que sea similar al evento de modificación, pero es absurdo pedir un arreglo para un único dato., en definitiva en ambos casos los implementadores tienen que hacer algo con el archivo sino no anda)
  • Los formulario_ml pueden trabajar con los ef_multi_seleccion
  • Cambios al ei_calendario, no funcionaba bien el cálculo del día viernes para una semana y año dados
  • Se eliminaron componentes transversales solo utilizados por el toba: objeto_esquema, objeto_esquema_db, objeto_grafico, objeto_plan, objeto_ut_multicheq
  • Nuevo objeto_ei_esquema (opcionalmente se necesita tener instalado GraphViz?)

Otros

  • (BUG) Los ítems públicos sin grupos de permisos asociados no se mostraban en el menú
  • (BUG) En la pantalla de login de los PA de toba se utiliza el mismo estilo que el proyecto
  • La función js::arreglo puede navegar arreglos de varios niveles
  • Foco automatico para textarea
  • Se encripta las claves de los usuarios en la base (#54).
  • Esquema básico de PERMISOS GLOBALES, los permisos del grupo actual se cargan en el inicio de sesion, se puede dejar que el sistema arme un mensaje o brindar uno personalizado. La violación del permiso lanza una excepcion aunque con un parametro puede retorna falso.
  • El mecanismo de autologin no envía la clave al cliente

Administrador

  • Documentación parcial en PHPDoc de los objetos nuevos
  • Este wiki esta disponible offline en la working copy local
  • Se pone border al frameset del admin para que se pueda hacer resize en firefox
  • El árbol de la operacion en el admin muestra un link para acceder a los eventos de un objeto.
  • Los test se integran con el code_coverage de xdebug y PHPUnit2, muestra la cobertura sobre los fuentes 'ejercitados', marcando de forma diferente las lineas por las que se testeo y de las que no
  • Se reescribe el editor de usuarios del admin para soportar la encriptación de las claves y a la vez soportar el formato no encriptado anterior (y de paso para hacer alguna operación con toba para ver como se siente).
  • El editor del datos_relación utiliza un ei_esquema para ir viendo la edición
  • Chequeo para que no se puedan asociar tablas sin asociar sus columnas
  • Administración de permisos globales.
