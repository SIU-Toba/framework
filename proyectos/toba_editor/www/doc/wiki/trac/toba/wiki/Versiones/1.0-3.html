Versiones/1.0.2

Versión 1.0.2

  • Fecha: 28/11/06
  • Fuente: source:tags/1.0.2
  • Revisiones involucradas: [2073] a [2170]

Lista de Cambios

Mejoras

  • Más funcionalidad para el proyecto toba_instancia
  • Inicio del tutorial en toba_referencia.
  • Nuevo tipo de solicitud para poder ejecutar archivos php planos que no tengan nada que ver con el esquema de solicitud_web. La solicitud_web puede ejecutar archivos php planos, pero dentro del esquema menu, javascript, tipo de pagina y demas, esto no es necesario para una salida en excel o svg o muchos otros formatos no-html, una opcion es usar otro servicio de la solicitud_web pero es sobrecargarla con algo que no tiene nada que ver.
  • Vuelve a andar el debug de los datos_relacion
  • Desde el punto de acceso de un proyecto se pueden ejecutar items de otros proyectos (que se tenga permiso), esto permite integrar proyectos en un mismo sistema. Se logra privilegiando el proyecto que viene por URL (ai) por sobre la constante apex_pa_proyecto que se usa solo cuando la url no explicita el proyecto (es como el proyecto por defecto)
  • Capacidad de sincronizar un dato para otra celda de memoria, asi es posible poner a otro proyecto en otra celda de memoria y darle parametros fuera de la URL
  • Nuevo método para retornar el path temp no-navegable
  • Nuevo metodo efs() en los forms js, para poder hacer algo asi, sin acceder a variables locales: 
     for (id_ef in this.efs()) {
    this.ef(id_ef).metodo();
 }
  • Modificacion en la logica de EXPORTACION de proyectos. Se granularizan los SQLs donde reside la exportacion de los GRUPOS de ACCESO. Se exportan los proyectos con este nuevo formato.
  • Se agregan 4 campos en apex_proyecto relacionados con el registro de la VERSION del proyecto.Esto campos fueron solicitados por Hernan Cobo para uniformar el manejo de versiones.
  • Es posible obtener por API los parametros de usuario

Bugs

  • Editor del datos_tabla: Al intentar leer los metadatos trataba de obtener la fuente de datos del editor y no del proyecto editado
  • El anexar datos del dt rompia los mapeos hechos durante el trabajo en memoria. Se agrega la posibilidad de auto-asignar padres en base a los cursores.
  • Bug en el form, la validacion de los efs cuando uno tiene cascada 'relajada' requiere que se registre el esqumea de cascadas, esto es en la etapa de eventos. Luego de la configuracion se requiere que se registre nuevamente por si algo cambio, como resultado se registra dos veces, al fin de la construccion y luego de la configuracion.
  • Bug en el ef_popup, por alguna extraña razón desde el commit [1523] el ef_popup carga su estado a partir de su descripcion si no tiene estado (!), supongo que seria por el esquema anterior de cascadas que usaba el estado del ef para responder, como ahora la carga de opciones la maneja el mismo form esto no es necesario. Este bug hacia que el popup no pudiera describir su estado null.
  • En la creacion del proyecto, se crea la fuente de datos pero no se ponia como predefinida
  • En la carga de opciones de efs, cuando venia null asumia que el master no estaba cargado, ahora esto se hace explicitamente
  • Cuando se ejecutaba un archivo SQL de varios MB y falla, la excepcion dejaba a php al 100% un largo rato... se opta por cortar la SQL que va en la excepcion.
  • Bug en la carga externa de filas en el datos_tabla: Si se usaba un dao de la misma clase los parametros hiban como un solo arreglo posicional. (Ver migración manual más abajo)

Migración Manual

La carga externa de campos de un datos_tabla a partir de un método php (dao) tenía un bug en la forma de pasar sus parámetros. La forma correcta es un parámetro por clave necesaria. Por ejemplo si la descripción de un campo depende de $id_cliente e $id_institucion, la función a definir tiene que ser por ejemplo: 

function get_descripcion($id_cliente, $id_institucion);

En los casos que la carga se realizaba a través de un método ubicada en la misma clase estos parámetros se encapsulaban en un arreglo posicional. Por eso es necesario:

  1. Buscar en el proyecto todas las ocurrencias de activar_proceso_carga_externa_dao
  2. Encontrar aquellas llamadas cuyos parámetros $clase e $include no son null (2do y 3er parámetro respectivamente). Esto indica que la carga se hace a partir de un método de la misma clase.
  3. Buscar el método encargado de la carga (1er parámetro del activar_proceso_carga_externa_dao)
  4. Este método seguramente asume que le llega un único parámetro y es un arreglo, remplazarlo por N parámetros explícitos según la definición de la carga.
  5. Reemplazar el cuerpo de la función según estos nuevos parámetros.
