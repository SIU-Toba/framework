Problemas Asociados a la forma en que PHP maneja los tipos de datos

Esto es algo molesto que oculto un bug un tiempo. La interface esperaba un array, pero estaba recibiendo un string. Esto hace pensar que hay que usar aserciones para robustecer un poco las interfaces. Este problema no ocurre si el indice del array es un entero o si $a es un entero. El problema es que "isset($a["xxx"])" devuelve un 1... ¿Alguien sabe porque? 

!#php
<?
	$a = "Este es un STRING";
	if(isset($a["xxx"])){
		echo "¿Por que entre aca?";
	}
?>

Al parecer hace una coersión de "xxx" a 0 (capaz es un bug del lenguaje), por que var_dump($a["xxx"]) (o de cualquier clave asociativa) retorna "E", que es $a{0}. Habría que preguntar is_array($a).

Llendo a las definiciones, PHP no es un lenguaje "No Tipado", ya que tiene tipos. Los 'problemas' son que:

  • Tiene "Tipado Dinámico", el tipo que toma una variable no se conoce hasta la ejecución: 
    <?php
    var $a;
    if (rand(0,1)) 
        $a = array();
    else
        $a = 8;
    print($a); //El tipo de $a sólo se conoce previo a la ejecución de la sentencia
?>
  • Es "No fuertemente tipado", esto quiere decir los errores de tipo no se detectan hasta la ejecución (hay varias definiciones de "fuertemente tipado" pero esta es la que mas me quedo!): 
    <?php
    $auto = $this->obtener_auto();
    $auto->manejar();    //No se puede asegurar que $auto contenga un metodo manejar
?>
  • Hace "Coersión de tipos", esto quiere decir que se hacen conversiones implicitas de los tipos de las variables, generalmente los lenguajes necesitan que se explicite la conversion (con funciones o haciendo casting), pero en php dependiendo de la operación se hacen coersiones: 
    <?php
    $a = "5" + 4;
    var_dump($a); //int(9)
    $entero = 1;
    $flotante = 0.5;
    $b =  $flotante + "8" + $entero;
    var_dump($b); //float(9.5)
?>
