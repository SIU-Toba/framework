Menu

Por un lado estan los items que forman parte y por otro el comportamiento gráfico del menú. ¿Se los podría llamar menu y tipo de menu?

Síntomas

La vista de "items" deberia perder su relacion con el menu que ve el usuario. Pasaria a ser una organizacion de elementos dedicada al desarrollador (Ver Modulos). (Esto esta relacionado con #49)

  • Se necesita un espacio para generar menues (publicar items organizados)
  • Deberia existir la posibilidad de generar un menu distinto para cada grupo de acceso.
  • Deberia existir la posibilidad de elegir el tipo de menu que se desea mostrar para cada grupo (deberia ser posible agregar tipos de menues).

Posibles Soluciones

  • Tipos de Menu: Para el comportamiento gráfico, en el branch de refactoring de includes se incluye un esquema inicial donde el tipo de menu es una propiedad del proyecto (tienen su tabla apex_menu) y el comportamiento esta en clases, el tipo de página es el encargado en definitiva de incluir el menú (es parte de la página). Los tipos de menus posibles están limitados por el layout del proyecto (con-frames o sin-frames).

Notas

  • S: Para la mayoría de los proyectos el arbol de desarrollo y el final de menu va a ser el mismo (es mucho mas comodo), no se debería perder eso de vista, permitir varios menus para un proyecto no tiene que complicarlo.
  • J: Si, completamente de acuerdo. Pero igual me suena que el menu deberia separarse de la organizacion del programador. Creo que hay dos opcciones que no son excluyentes entre si para facilitar esto:
    1. Mostrar por defecto la estructura de modulos si el usuario no definio una vista particular.
    2. El lugar del administrador donde se "publican" los items para un grupo de acceso puntual deberia tener una opcion para generar automaticamente el arbol en base a la estructura de modulos.
