Inquietudes de Desarrollo

Hace unos días surgio una propuesta para hacer una reunión entre consultores del SIU para fomentar la transferencia de conocimientos en el proceso de Desarrollo de SW (también podría llamarse ingeniería de SW pero quizás es un término un poco más amplio).

Cosas Incluídas

Este es el listado de cosas que nos inquietan:

Codificación

  • ¿Hay separación entre lógica y presentación? ¿De qué forma? Por ejemplo hacer que un listado o un formulario tenga un layout totalmente distinto al original ¿implica modificar código cercano al que pone la lógica?.
  • ¿Se usa Programación Orientada a Objetos?
    • Escenarios de aplicación de herencia
    • Utilidad de los 'patrones de diseño'
  • ¿Hay estandares de nomenclatura?
  • ¿Hay documentación incluída en el código? ¿Se puede consultar sin visualizar el mismo? ¿Esta actualizada?
  • Información estadística, si hay, al estilo líneas de código sin comentarios, cantidad de clases o funciones.

Diseño - Arquitectura

  • Suponiendo que el sistema tiene un tiempo de vida largo en donde existen 3eros encargados de generar personalizaciones ¿Hay algún mecanismo estandar soportado?
  • ¿Cómo se encara el problema de bajas lógicas de registros?

Testing

  • ¿Se asegura de alguna manera que las ultimas modificaciones no rompan alguna funcionalidad anterior?
  • ¿Cuando se dice que una funcionalidad está lista?

Metodologías

  • Los desarrolladores trabajan en equipo, sobre los mismos módulos o cada cual tiene un sector bien delimitado.
  • Asumiendo que participan varios desarrolladores ¿Cómo se coordinan las modificaciones al código?

Cosas NO incluidas

Para aclarar un poco el marco de las inquietudes, se listan las cosas que giran alrededor del desarrollo pero no son centrales para el caso:

  • Análisis de requerimientos
  • Creación de especificaciones
  • Comunicación con el cliente
  • Managment del proyecto
