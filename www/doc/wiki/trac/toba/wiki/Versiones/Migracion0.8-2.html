Instructivo de Migración hacia 0.8.3

ATENCION: Cada versión de toba tiene su propio esquemas de includes javascript, por lo que es necesario asegurarse que el browser no los cachea, o bien hay que borrar el cache o directamente desactivarlo (recomendado).

La migración se divide en cambios a los metadatos y cambios al código.

Metadatos

Antes de actualizar el código de toba y convertir los proyectos es recomendable hacer una exportación de metadatos 

proyecto exportar mi_proyecto

y subirlo al svn. Para mantener los usuarios actuales de los proyectos es necesarios exportalos al sistema de archivos: 

instancia exportar_local

Luego, una vez hecho el switch al tag, es necesario regenerar la instancia: 

instancia regenerar

Para aquellos proyectos ubicados en la versión 0.8.2 se provee un par de scripts de conversión. Una vez actualizado a esta versión de toba se puede averiguar que conversiones son posibles sobre un proyecto particular: 

conversion listar mi_proyecto

A continuación se deben correr las actualizaciones correspondientes a esta versión 

conversion ejecutar mi_proyecto 0.8.3
conversion ejecutar mi_proyecto 0.8.3.1
conversion ejecutar mi_proyecto 0.8.3.3

Código

Puntos de acceso

El cambio a los puntos de acceso se debe a la necesidad de incluir al nuevo núcleo de toba. También es recomendado sacar del php.ini el include_path de toba y dejar que el punto de acceso lo determine solo (esto permite tener varias instalaciones de toba en una máquina). El código que hace esto es: 

################################################################################
if (isset($_SERVER['TOBA_DIR'])) {
	$dir = $_SERVER['TOBA_DIR']."/php"; 
	$separador = (substr(PHP_OS, 0, 3) == 'WIN') ? ";.;" : ":.:";
	ini_set("include_path", ini_get("include_path"). $separador . $dir);
}
require_once("nucleo/toba.php");
toba::get_nucleo()->acceso_web();	
################################################################################

en reemplazo del viejo: 

require_once("nucleo/acceso_browser.php");

Se pueden actualizar automáticamente los puntos de acceso de un proyecto con el comando: 

proyecto conf mi_proyecto

Parametro de los EVENTOS del componente ei_cuadro

Los eventos disparados por el componente ei_cuadro proveen como parametro el valor de las columnas definidas como ID en la fila que fue seleccionada. En la version anterior, este valor era un string si la clave era simple y o un array asociativo si la clave era compuesta (EJ: Si el ID estaba formado por los campos 'anio' y 'mes', el parametro tendria el formato: array('anio'=>2005, 'mes'=>11); ). En la version actual se utiliza un array asociativo en todos los casos. Por ejemplo el siguiente fragmento de codigo cambia su comportamiento en la nueva version. 

function evt__cuadro__seleccion($id){
$this->seleccion = $id;
}

Para lograr el mismo comportamiento en la version actual, y suponiendo que la clave definida en el cuadro se llama 'identificador' hay que escribir los siguiente. 

function evt__cuadro__seleccion($id){
$this->seleccion = $id['identificador'];
}

Utilizacion del metodo consultar_fuente($sql, $fuente)

En la version anterior si la consulta no devolvia datos retornaba NULL, ahora devuelve un ARRAY VACIO. Por ejemplo el siguiente fragmento de codigo cambia su comportamiento en la nueva version. 

$a = consultar_fuente($sql);
if(isset($a)){
echo "Hay datos";
}else{
echo "No hay datos";
}

El siguiente codigo seguiria funcionando igual 

$a = consultar_fuente($sql);
if( $a ){
echo "Hay datos";
}else{
echo "No hay datos";
}

Para poder llevar a cabo una migracion ordenada, se provee el metodo consultar_fuente_obs() que posee el mismo comportamiento que la funcion anterior. Este cambio puede hacerse como un reemplazo masivo de texto.

Alto y ancho de componentes

Una de las implicaciones de la utilizacion del DOCTYPE HTML (#38) es que la definicion de los anchos y altos tiene que estar acompañada de la unidad de medida utilizada (Por ejemplo '%' o 'px'). Si se utilizaron valores absolutos, hay que agregarle el sufijo 'px'.
